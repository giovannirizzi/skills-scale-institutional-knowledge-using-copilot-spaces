# OctoAcme — New Team Member Onboarding Checklist

## Purpose
Ensure that every new project participant — regardless of role — is set up for success from day one. Use this checklist when onboarding any new member to an OctoAcme project.

---

## General Onboarding (All Roles)

- [ ] Access to the project repository and project board granted
- [ ] Added to relevant communication channels (Slack, Teams, etc.)
- [ ] Introduced to the team during a standup or kickoff meeting
- [ ] Shared links to key project docs (One-pager, Planning doc, Roles & Personas)
- [ ] Walkthrough of project board columns and workflow conventions
- [ ] Definition of Done reviewed and understood
- [ ] Escalation paths and communication cadence explained

---

## Role-Specific Onboarding

### Developer
- [ ] Development environment setup guide followed
- [ ] Branching and PR conventions reviewed
- [ ] CI/CD pipeline walkthrough completed
- [ ] Introduced to code owners and review process

### Designer
- [ ] Access to design tool (e.g., Figma) and shared design system granted
- [ ] Reviewed existing wireframes, user flows, and style guides
- [ ] Met with Product Manager to understand user goals and current priorities
- [ ] Handoff workflow with Developers explained

### QA/Test Engineer
- [ ] Access to test environment and test management tools granted
- [ ] Existing test plans and automation suite reviewed
- [ ] Defect triage and reporting process explained
- [ ] Release sign-off process and criteria reviewed

### Data Analyst
- [ ] Access to analytics platform and data sources granted
- [ ] Existing dashboards and reports reviewed
- [ ] Success metrics for current sprint/milestone reviewed with Product Manager
- [ ] Instrumentation and tagging conventions documented

### Security/Compliance Stakeholder
- [ ] Briefed on project scope and any known compliance requirements
- [ ] Access to security tooling and CI security scan results granted
- [ ] Security review and sign-off process documented and shared
- [ ] Incident response runbook location shared

### Support/Customer Success
- [ ] Access to support ticketing system granted
- [ ] Upcoming release roadmap and change log reviewed
- [ ] Bug escalation path to Developers and Product Managers established
- [ ] Customer feedback channels and feedback loop process explained

### Executive Sponsor
- [ ] Project One-pager and success metrics reviewed
- [ ] Milestone review calendar shared
- [ ] Escalation path to/from Executive Sponsor documented
- [ ] Preferred communication format (email, briefing doc, dashboard) agreed upon
